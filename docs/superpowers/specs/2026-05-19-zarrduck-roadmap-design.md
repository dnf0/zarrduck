# Zarrduck Future Roadmap Design

**Date:** 2026-05-19

## 1. Context & Purpose
The `zarrduck` CLI has achieved its initial goal of becoming an agentic spatial data engine capable of extracting and inspecting GeoZarr arrays. To elevate it to the forefront of geospatial workflows, it needs to solve three critical industry pain points: discovering data, performing temporal aggregations, and converting legacy data formats into cloud-native Zarr.

This document outlines a phased roadmap to implement these advanced capabilities.

## 2. Phase 1: STAC Discovery (`zarrduck search`)
Currently, users must manually locate Zarr URIs from external catalogs.

### Architecture
- Introduce the `stac` crate to interact with SpatioTemporal Asset Catalog (STAC) APIs.
- Add a new `search` command to the CLI: `zarrduck search --api <url> --collection <name> --bbox <w,s,e,n> --datetime <range>`.
- The command will query the STAC API, parse the resulting JSON items, and extract the `href` assets pointing to Zarr stores.
- Output can be formatted as a table for humans or pure JSON for agents, allowing seamless chaining into the `extract` command.

## 3. Phase 2: Temporal Analytics (`zarrduck resample`)
Geospatial data is typically time-series data. Aggregating this data (e.g., daily to monthly means) is currently tedious in pure SQL.

### Architecture
- Add a new `resample` command that operates on the local `.duckdb` files generated by the `extract` command.
- Command signature: `zarrduck resample analysis.duckdb --time-col time --freq month --agg mean --out monthly.duckdb`.
- The CLI will dynamically generate DuckDB SQL using `date_trunc()` and `GROUP BY` to perform the temporal aggregation efficiently.
- This command abstracts away the complex SQL windowing and grouping required for time-series geospatial analysis.

## 4. Phase 3: Data Ingestion Engine (`zarrduck ingest`)
Users often possess raw NetCDF or GeoTIFF files and need to convert them into optimized GeoZarr format on cloud storage.

### Architecture
- Add a new `ingest` command: `zarrduck ingest local.nc s3://bucket/out.zarr`.
- This feature will heavily leverage DuckDB's `spatial` extension (which wraps GDAL) to read the legacy raster formats.
- The CLI will determine the optimal chunk shape, set up the GeoZarr metadata (affine transforms, CRS), and stream the data from DuckDB into Zarrs, utilizing our existing export engine logic.
- This effectively turns `zarrduck` into a cloud-native ETL pipeline engine.

## 5. Development Strategy
These phases are modular and will be implemented sequentially.
1. Design and implement the `search` command.
2. Design and implement the `resample` command.
3. Design and implement the `ingest` command.
