# Eider CLI Design

**Date:** 2026-05-18

## 1. Context & Purpose
The current `geozarr-cli` provides a basic interface for executing raw SQL against DuckDB and exporting the results back to Zarr. However, passing raw, complex SQL queries is too shallow an abstraction. Users (and LLM agents) require a deeper tool that abstracts away the complexities of spatial SQL, projection math, and chunk-level pruning.

The purpose of this project is to rename the CLI to **`eider`** and elevate it into a first-class, self-documenting spatial operations engine. Its core value proposition is seamlessly bridging the gap between cloud-native raster data (Zarr) and local vector data (polygons), allowing users to isolate data and drop into an interactive analytical environment.

## 2. Core Architecture & Commands

The `eider` CLI will be built using Rust's `clap` crate to ensure extensive, auto-generated documentation and examples.

### 2.1 Discovery: `eider info`
- **Purpose:** Agent-friendly discovery of remote Zarr datasets.
- **Input:** A Zarr URI (e.g., `s3://bucket/climate.zarr`).
- **Output:** Clean JSON (or an ASCII table for humans) detailing the array shape, chunk shape, data types, and parsed GeoZarr CRS metadata.

### 2.2 The Killer Feature: `eider extract`
- **Purpose:** Vector-Raster integration (Zonal Extraction). Extracts Zarr raster data constrained by vector polygons.
- **Input:** A Zarr URI, a vector file (e.g., `boundaries.geojson`), and an output DuckDB database path (`--out analysis.duckdb`).
- **Data Flow:**
  1. Parse the vector file to determine the global bounding box.
  2. Generate optimized DuckDB SQL that uses named parameters (`lat_min`, `lat_max`, etc.) to strictly prune Zarr network requests to the bounding box.
  3. Load the spatial vector data into DuckDB.
  4. Perform a spatial join (e.g., `ST_Contains`) between the polygon geometries and the projected Zarr coordinates (`lon`, `lat`).
  5. Materialize the extracted, flattened data into a persistent table within the output `.duckdb` file.

### 2.3 Interactive Analytics: `eider shell`
- **Purpose:** Drop the user or agent into an interactive environment for further manipulation.
- **Input:** A local `.duckdb` file (typically generated by `extract`).
- **Data Flow:** Spawns an interactive DuckDB REPL session pre-loaded with the `spatial` and `eider` extensions.

## 3. Agent-First Design Principles
- **No Boilerplate:** Progress bars and ANSI colors must be cleanly disabled when the output is redirected or when an explicit `--json` flag is provided.
- **Self-Documenting:** Extensive `long_about` descriptions in the CLI help text will provide copy-pasteable examples for LLM agents to learn the tool's syntax instantly.

## 4. Development Strategy
1. Rename the existing `cli` crate to `eider`.
2. Implement the multi-command `clap` structure.
3. Build the `info` command by leveraging the new `read_zarr_metadata` table function.
4. Build the `extract` command, integrating with DuckDB's spatial extension for the vector-raster join.
