Performance Benchmarks

Two kinds of numbers live here. Measured results are single-machine criterion microbenchmarks of operations inside the extension — they show where time goes. Head-to-head is a controlled, correctness-gated comparison of the zonal-statistics kernel against the established raster-zonal stack (exactextract, rasterstats). Run either yourself with the commands in Reproducing these numbers.

Measured results​

BenchmarkWhat it measuresMedian
populate_lat_batch_2048Generating 2,048 coordinate values from an affine transform (no network)~10.6 µs
scanner_read_chunk_subset_remoteReading a chunk subset through the scanner's remote (full-chunk) read path, backed by an in-memory store (no network)~3.4 µs

Coordinate generation is essentially free relative to I/O. Because Eider computes lat/lon from the array's affine transform on the fly (translation + index × scale) rather than fetching coordinate arrays, a full batch of 2,048 values costs microseconds — negligible next to the network round trips that dominate a remote read. See Spatial Pruning for how that transform is also used to prune chunks.

Head-to-head: eider vs the raster-zonal stack​

How does eider's read-then-DuckDB-spatial-join approach compare to the purpose-built raster-zonal tools — exactextract (C++, exact coverage-fraction area weighting) and rasterstats (the Python baseline) — on the compute kernel?

To isolate the kernel, every tool reads the same warm, local Cloud-Optimized GeoTIFF (so eider's separate cloud-native partial-read advantage — pruning chunks from remote Zarr/COG/STAC, see Spatial Pruning — is deliberately out of scope here; this is the fight on exactextract's home turf). Synthetic data, both asset/cell regimes, correctness verified before any timing is reported.

Single machine (Apple arm64), DuckDB 1.5.2, exactextract 0.3.0, rasterstats 0.21.0; synthetic gradient field; median of 3 reps (rasterstats: 1 rep — see below), warmup excluded but the OS file cache is warm. Each tool's read of the COG is inside its timed region — including eider's read_geo + per-cell geometry build. Treat seconds as order-of-magnitude (run-to-run variance on the minute-scale cells is real). rasterstats capped at 20k polygons (IMPRACT above that).

Correctness first​

Where conventions match, the tools agree to within floating-point noise — this is the precondition for comparing speed at all:

Convention / metriceider vs referencemax abs diff
centroid / maxvs rasterstats (center-rasterize)0 (exact)
all-touched / maxvs rasterstats & exactextract0 (exact)
all-touched / meanvs rasterstats~1.0e-5
area-weighted / meanvs exactextract (exact coverage)~1.2e-8

The one disagreement the gate surfaced is rasterstats on sub-pixel footprints: its GDAL all_touched rasterizer selects a slightly different cell set than the exact geometry (diffs of 1.7–4.4 on a 0–50 field). eider matches exactextract — the exact coverage reference — so eider's geometry path is the correct one; rasterstats is the outlier there.

One asymmetry to keep in mind for the all-touched rows: exactextract has no "touched, unweighted" mode — it always computes coverage fractions for every intersecting cell, so for all-touched it does strictly more numerical work than eider's boolean ST_Intersects predicate. The cleanest same-work-both-sides comparison is therefore area-weighted mean (both compute coverage); the all-touched/centroid gaps partly reflect exactextract doing extra work, not pure engine speed. (exactextract is absent from the centroid rows entirely — it has no cell-centre-membership mode.)

Regime 1 — sub-cell assets (point sample over a coarse grid)​

Convention / metricneider (spatial)eider (index-join)exactextractrasterstats
centroid / max1M0.420.38—IMPRACT
area-weighted / mean1M9.96—93.0n/a

When the asset sits inside one cell, modelling it as a point and using an arithmetic cell-index equi-join is the right tool — nothing else is close (0.38 s / 1M). This is the only regime where that shortcut is both correct and fastest. (all-touched is omitted here: on these sub-pixel polygons rasterstats disagrees with the exact geometry — see Correctness first — so a timing race there would not be comparing equal results.)

Verdict (honest)​

  • Centroid and all-touched, every scale: eider/DuckDB is fastest — roughly 6–8× exactextract and ~25–1000× rasterstats, with identical results. Caveat: part of the exactextract gap is the asymmetry above (it computes coverage even when only "touched" is asked). The enabling trick is precomputing the cell point/box as columns so DuckDB's spatial join engages — inline geometry in the join predicate does not, and blows up (a real footgun we hit and fixed).
  • Area-weighted mean — the fairest comparison (both compute coverage) — eider stays ahead but it's closest: ~1.6× at 100k (7.0 s vs 11.2 s) and ~1.5× at 1M many-cell (69 s vs 107 s); larger when footprints cover few cells (Regime 1 1M: 10 s vs 93 s). These are the same order of magnitude — exactextract is a mature, robust C++ implementation and a fine choice here; eider's edge is modest, not a blowout.
  • rasterstats is correct for centroid but ~1000× slower (not viable at scale) and diverges on sub-pixel footprints.

So for these conventions over many polygons, eider/DuckDB was the fastest correct option in every comparable cell measured — decisively for centroid/all-touched, modestly for area-weighted. And remember this hands every tool a warm local raster: eider's other advantage — never downloading the cells you don't need, from remote Zarr/COG/STAC — is a separate axis not measured here.

Remote partial reads: how much does eider actually fetch?​

The head-to-head above gave every tool a warm local raster to isolate compute. This section measures eider's other advantage — only fetching the chunks a query needs over the network — against a naive whole-array read and a competent chunk-aware baseline (xarray/zarr .sel; rasterio windowed /vsicurl), which also prune. The honest question: does eider's pushdown actually cut bytes, and does it hold for Zarr v3 (incl. sharded), the production format?

Method: a generated ~4000×4000 store (~51 MB) served from a local, Range-capable, byte-logging HTTP server; per query window we record bytes fetched + HTTP requests (deterministic) — wall time is localhost-only and a footnote. Every cell passes a correctness gate (all three return identical window values) before bytes are reported.

Zarr (v2, v3, sharded) and COG — eider prunes like the best chunk-aware client​

Bytes fetched as a fraction of the full store (÷ naive); a tiny (≤1%-area) window:

formateiderchunk-awarenaive
Zarr v20.79 MB (0.017×)0.79 MB46 MB
Zarr v30.88 MB (0.017×)0.88 MB51 MB
Zarr v3 (sharded)0.88 MB (0.017×)0.88 MB51 MB
COG0.84 MB (0.017×)0.84 MB51 MB (3,111 reqs, 12 s)

eider fetches ~1.7% of the store — essentially identical to a hand-written xarray.sel / rasterio window, and ~60× less than naive. Zarr v3 prunes exactly as well as v2, and sharded v3 prunes perfectly to inner chunks over HTTP using ranged partial reads. (At a 10%-area window all four formats rise to ~0.147× — you genuinely need more chunks.) eider's edge over the baselines is ergonomic: the window is a bbox in SQL, uniform across Zarr/COG/STAC, with no hand-computed index ranges.

Verdict​

  • Zarr (v2, v3, sharded) and COG: eider's pushdown is best-in-class — ~1.7% of the store for a small window, matching a competent chunk-aware client and ~60× below naive, from a bbox in SQL. (See Spatial Pruning.)
  • Sharded v3 over HTTP: eider uses ranged reads to resolve the shard index and fetches only the requested inner chunks, working out of the box with the spec-default index_location:"end".
  • Bytes are the deterministic story; localhost time is incidental (the real-world win grows with network latency, since fewer bytes = fewer/again-smaller round trips).

Reproducing these numbers​

The criterion microbenchmarks live in the repo:

# Coordinate generation (extension crate)
cargo bench --bench coordinate_bench

# Remote chunk-subset read (geozarr_core crate)
cargo bench --package geozarr_core --bench scanner_bench

Sources: extension/benches/coordinate_bench.rs and geozarr_core/benches/scanner_bench.rs. Numbers vary with hardware, network, and the remote store; treat them as order-of-magnitude, not absolute.

The head-to-head harness is a standalone Python script:

python3 -m venv /tmp/bench_venv && source /tmp/bench_venv/bin/activate
pip install -r scripts/bench_requirements.txt
# build the eider extension if needed, then:
export GEOZARR_ALLOW_PATH="$PWD"
python scripts/bench_zonal_headtohead.py --regime both \
  --counts 10000 100000 1000000 --reps 3 --budget-seconds 150 --rasterstats-max-n 20000 \
  --json bench_full.json

It generates the synthetic COG + polygons, runs the correctness gate (timing is reported only after it passes), and prints the matrix. Source: scripts/bench_zonal_headtohead.py. See Zonal statistics for the convention definitions.

The remote partial-read harness (bytes fetched over a local HTTP server, across Zarr v2/v3/sharded + COG):

source /tmp/bench_venv/bin/activate   # same venv (needs zarr>=3)
export GEOZARR_ALLOW_PATH="$PWD"
python scripts/bench_remote_partialread.py \
  --formats zarr_v2 zarr_v3 zarr_v3_sharded cog \
  --windows 0.001 0.01 0.1 --shape 4000 4000 --reps 3 --json remote_results.json

Source: scripts/bench_remote_partialread.py. Each cell passes a correctness gate before its bytes/requests are reported.
